# Redis Desktop Native Titlebar Theme Design

**日期**

- 2026-03-24

**背景**

- 当前应用内容区主题切换已经生效，但 macOS 原生标题栏仍保持系统浅色外观。
- 这会导致深色主题下窗口最上方出现一条突兀的白色区域，破坏整体一致性。
- 启动阶段已经读取了设置，但没有把已保存主题用于原生窗口初始化，存在首屏外观割裂的风险。

**目标**

- 让 macOS 原生标题栏在手动亮色/暗色主题下与应用主题一致。
- 在“跟随系统”模式下继续遵循系统外观，不引入额外状态分叉。
- 尽量只补窗口壳层同步，不改现有内容区布局和主题桥接结构。

**结论**

- 保留原生标题栏和 traffic lights，不做自绘无边框方案。
- 新增“主题偏好 -> 原生窗口主题”的映射函数，统一用于启动阶段和运行时切换。
- 应用启动时先按已保存主题初始化窗口外观；运行时再通过桌面窗口句柄同步 `set_theme`。

## 一、方案选择

- 不采用完全自绘标题栏。当前收益不够高，且会引入拖拽、缩放、焦点态、全屏和可访问性成本。
- 不在本次改动中引入透明标题栏和内容延伸。那是下一层视觉优化，不是解决当前问题的最小改动。
- 采用原生标题栏明暗同步，优先解决“白色标题栏与深色内容断层”问题。

## 二、实现策略

`src/theme/colors.rs`

- 新增原生窗口主题映射函数。
- `ThemePreference::System` 返回 `None`，表示继续跟随系统。
- 手动主题根据 `ThemeSpec.kind` 映射到 `tao::window::Theme::Light/Dark`。

`src/main.rs`

- 读取已保存设置后，将窗口 builder 的初始 `theme` 设为映射结果。
- 这样可以减少窗口创建后的标题栏闪白问题。

`src/ui/app.rs`

- 将初始 `theme_preference` 前移到 signal 初始化阶段，减少首屏内容区和窗口壳层不一致。
- 使用 Dioxus 的窗口句柄在主题偏好变更时调用原生 `set_theme`。

## 三、验收标准

- 手动切到任一暗色主题后，macOS 标题栏不再保持白色。
- 手动切到亮色主题后，标题栏恢复亮色原生外观。
- 跟随系统模式下，标题栏继续由系统控制。
- `cargo check` 通过，无新增类型错误。
