# Redis Desktop Dialog Titlebar Unification Design

**日期**

- 2026-03-25

**背景**

- 当前多个弹窗都复用了 `AnimatedDialog`，但标题节点仍然各自写在内容区里。
- 关闭按钮已经统一为右上角按钮后，标题却继续由各弹窗自己决定位置、字号、颜色和间距，导致按钮与标题基线不一致。
- 受影响最明显的是设置、新建连接、新增 Key、内存分析、按模式删除、设置 TTL 这几类高频弹窗。

**目标**

- 为桌面弹窗定义单一且统一的标题栏骨架。
- 让标题文字和右上角关闭按钮在所有目标弹窗里都使用同一套对齐规则。
- 让弹窗正文从统一的标题栏下边界开始布局，不再允许目标弹窗自行放置标题。

**结论**

- 将标题栏沉入 `AnimatedDialog`，由容器直接负责标题、按钮、标题区留白和正文起始基线。
- 目标弹窗只传入纯文本标题，不再在内容区手写 `h2/h3`、图标或 emoji。
- 统一标题栏采用单行结构：左侧标题文字，右侧关闭按钮，中间保留弹性空间，标题区高度和左右内边距固定。

## 一、方案选择

- 不采用“继续保留各弹窗自定义标题，只是人工对齐”的方案。那只能修当前页面，不能阻止后续继续漂移。
- 不采用“额外新建一个 Header 组件，但仍由弹窗自己放置”的方案。布局责任仍然散落在调用方，长期还是会失控。
- 采用“容器内建标题栏”的方案，把标题对齐和关闭按钮关系一次性统一。

## 二、实现策略

`src/ui/animated_dialog.rs`

- 新增可选 `title` 参数。
- 当传入 `title` 时，弹窗面板切换为“标题栏 + 内容区”的双层结构。
- 标题栏固定左右内边距、标题字号、底部分隔和关闭按钮位置。
- 未传入 `title` 的旧弹窗保持兼容，继续使用当前无标题结构。

目标弹窗迁移范围：

- `src/ui/settings_dialog.rs`
- `src/ui/connection_form.rs`
- `src/ui/add_key_dialog.rs`
- `src/ui/memory_analysis_dialog.rs`
- `src/ui/pattern_delete_dialog.rs`
- `src/ui/batch_ttl_dialog.rs`

迁移原则：

- 删除内容区内原有标题节点。
- 统一改为向 `AnimatedDialog` 传入纯文本标题。
- 不在本次改动中保留标题图标、emoji、强调色或副标题，保证统一性优先。

## 三、验收标准

- 目标弹窗标题文字与关闭按钮在视觉上使用同一标题栏基线。
- 同一类弹窗打开时，正文内容都从相同的顶部起始位置开始布局。
- 不出现标题重复、标题与关闭按钮重叠、正文贴到标题栏的问题。
- `cargo check` 通过；若有工作区内既有未提交改动导致的独立编译错误，需要明确指出。
